# CLAUDE.md — Опрос ПУ (аналитика собираемости приборов учёта)

Веб-приложение: принимает выгрузку «Опрос ПУ» из ПО «Пирамида» (xlsx, ~130 тыс. строк,
12–15 МБ), ведёт реестр приборов учёта (ПУ) и показывает собираемость (% опроса)
в разрезах. Задеплоено на Амвере, встраивается iframe'ом в оболочку «ПО Платформа»
(https://github.com/ashinoff/SUE_system).

Общаться с пользователем и писать комментарии/интерфейс — на русском.
Домен: энергетика, Сочинские электросети. Пользователи — не программисты.

## Стек и запуск

- Python 3.12, FastAPI, SQLAlchemy 2, PostgreSQL (prod, CNPG на Амвере) / SQLite (локально)
- Фронт: SPA на vanilla JS + Chart.js (CDN), файлы в `static/`, без сборки
- Excel-экспорт: xlsxwriter
- Локальный запуск: `DATA_DIR=./data python -m uvicorn app.main:app --port 8000`
  (без DATABASE_URL работает на SQLite — удобно для отладки)
- Prod: Dockerfile + `amvera.yml` (containerPort 80, persistenceMount /data)

### Переменные окружения
| Переменная | Назначение |
|---|---|
| `DATABASE_URL` | prod: `postgresql://Ashinoff:...@amvera-ashinoff-cnpg-oprosdb-rw:5432/Oprosdb` — регистр имён важен! |

## Структура кода

```

  parser.py    — потоковый разбор xlsx (openpyxl read_only), классификаторы
  importer.py  — импорт загрузки в фоне, дифф с предыдущей, автозакрытие заданий
  analytics.py — все отчёты (сводка, СПОДЭС, маршруты, мёртвые ТП, приоритеты...)
```

## Формат входного файла (стабилен, из Пирамиды)

Лист «ПОДРОБНО». Строки 1–9 — шапка (период, итоги), строка 10 — заголовки,
данные с 11-й. Парсер ищет лист по подстроке «ПОДРОБ» и строку с 'ПУ','Тип ПУ'.

Колонки (0-индексация в row):
- B(1) ПУ, C(2) Тип ПУ («(СПОДЭС)» в названии → is_spodes; производитель — по префиксу,
  правила в `parser.VENDOR_RULES`), D(3) Серийный номер
- E(4) Фазность, F(5) Маршрут опроса → `classify_route`: МКС / RootRouter / RTR /
  GSM (телефоны) / IP-шлюз (голые IP) / Без порта / Прочее; часть до « - » = ведущее устройство
- G(6) Дата опроса, H(7) Точка учёта (адресный путь — КЛЮЧ для логики «отремонтировано»)
- I(8) Область, J(9) **РЭС** (8 шт.: Адлерский, Сочинский, Туапсинский, Дагомысский,
  Хостинский, Лазаревский, Краснополянский, ПС РЭС)
- K(10) **Фидер**, L(11) **ТП** ← внимание: ТП именно в L, не в K!
- M(12) Тип абонента (ФЛ/ЮЛ/НЕТ ПРИВЯЗКИ), N(13) Тип ТУ, O(14) Отключен
- P(15) **Собираемость** («Собирается» / пусто), Q(16) Угасание сбора,
  R(17) Модуляция, S(18) Тип данных
- Строки могут быть короче 19 ячеек — парсер дополняет None; дубли ключа внутри
  файла пропускаются (первая строка побеждает)

Ключ ПУ в реестре: `meter_key = Тип ПУ || Серийный` (уникальный в Meter).

## Бизнес-логика (важно не сломать)

- **Загрузка = снимок.** Upload хранит итоги, MeterState — состояние каждого ПУ
  в этой загрузке. Meter — «сухой остаток» (реестр) с active/replaced_by.
- **Дифф** (importer.py): added / removed / collect_lost / collect_restored / moved.
- **«Отремонтировано»**: на той же Точке учёта (колонка H) старый ПУ исчез, появился
  новый с другим серийником, и он собирается → `repaired`; если не собирается →
  `serial_changed`. Пары строятся zip'ом выбывших/добавленных по tu_path.
- **Мёртвая ТП** = на ТП ≥2 ПУ и ни один не собирается → «не работает ведущее
  устройство» (показываем route_device).
- **Приоритеты для участков**: 1 — мёртвые ТП, 2 — угасание сбора (кол. Q),
  3 — одиночные ПУ без опроса не на мёртвых ТП.
- **Задания** (Task): создаёт staff/admin/uploader, авто-закрываются при импорте,
  если ПУ задания начал собираться.
- **Роли**: admin (всё), uploader (всё кроме удаления БД и пользователей),
  staff — служба учёта (видит всё, задания, не грузит), res — участок (ТОЛЬКО свой
  РЭС — принудительно в `auth.visible_res` + `_resolve` в main.py, фильтр на бэке,
  не доверять фронту).
- Обработка файла — фоновый поток, статус processing/done/error, фронт поллит
  `/api/uploads`. Параллельный импорт запрещён (409).

## Тестирование

- Реальный файл ~130 728 ПУ импортируется за ~100 сек (SQLite), итоги должны
  сходиться с шапкой файла (строки 4–8: всего/в опросе/без опроса).
- Для диффа есть приём: скопировать реальный файл и внести контролируемые правки
  (замена серийника на том же адресе ± опрос, удаление строк, новые строки) —
  проверить типы изменений в `/api/report/changes`.
- Быстрый smoke: логин admin → загрузка → summary → export/summary (xlsx открывается).

## Интеграция с Платформой (SUE_system, Keycloak SSO)

Приложение зарегистрировано в реестре Платформы (`SUE_system/src/config/apps.js`,
id `opros`, иконка `OprosTile` в `appIcons.jsx`). Домен Amvera:
`opros-piramida-ashinoff.amvera.io`.

**Схема (как у СИЗ/Светлячка):** Keycloak решает «кто ты» (по **email**) + «пускать
ли» (realm-роль **`opros-user`**). Приложение решает «что можно» — роль/РЭС из своей
БД по email, НЕ из токена. Один пользователь в Keycloak = одна роль доступа + email.

- Бэк: `app/config.py` (env), `app/keycloak.py` (проверка токена по JWKS: подпись/iss/
  exp/azp==web-desktop, aud не требуем), `auth.resolve_platform_user()` (гейт `opros-user`,
  связка по keycloak_id→email), эндпоинт `POST /api/auth/platform` (обмен на свою сессию,
  формат как `/api/login`). CSP `frame-ancestors` — middleware в `main.py`.
- Фронт (`static/app.js`): слушает `postMessage {type:'platform-auth',token}` ТОЛЬКО от
  `PLATFORM_ORIGIN`, шлёт `{type:'app-ready'}` родителю, меняет токен на сессию, fallback
  на обычный вход. Токен сессии хранится в `localStorage(opros_token)` и шлётся как
  `Bearer` (чтобы работать во встроенном iframe, где Lax-cookie не уходит на XHR).
- Env на Amvera для включения: `PLATFORM_SSO=true` (KEYCLOAK_ISSUER/JWKS_URL/AZP,
  PLATFORM_ORIGIN, OPROS_ACCESS_ROLE имеют дефолты под наш realm). БД: у User добавлены
  `email`, `keycloak_id` (лёгкая миграция `_ensure_user_columns` в `db.py`).
- В учётках приложения (вкладка «Пользователи») проставить email = email в Keycloak.
