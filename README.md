# Опрос ПУ — аналитика собираемости приборов учёта

Сервис принимает выгрузку «Опрос ПУ» из Пирамиды (лист «ПОДРОБНО», ~130 тыс. строк,
12–15 МБ), ведёт базу приборов и показывает собираемость в разрезах.
Работает как отдельное приложение и встраивается окном в оболочку Платформы
(SUE_system) через iframe — фронт уже слушает `postMessage {type:'platform-auth', token}`.

## Что умеет (MVP)

- **Сводка**: по каждому РЭС — всего ПУ / собирается / не собирается / % опроса,
  таблица с датами загрузок (факт) и графики динамики.
- **СПОДЭС** отдельно: количество и % по РЭС и по типам ПУ.
- **Маршруты**: через что опрашивается (МКС / RootRouter / RTR / GSM / IP-шлюз / без порта),
  разбивка по производителям (по началу наименования типа ПУ), работоспособность
  ведущих устройств (0 из N — «НЕ РАБОТАЕТ», <50% — «Деградация»).
- **Неисправные ТП**: если на ТП не опрашивается ни один ПУ — вывод «не работает
  ведущее устройство», с идентификатором устройства из маршрута.
- **Приоритеты для участка**: 1 — мёртвые ТП, 2 — угасает сбор, 3 — одиночные без опроса.
- **Изменения** между загрузками: добавлено / выбыло / заменён ПУ на том же адресе
  (если опрос появился — «отремонтировано») / потеря и восстановление опроса /
  смена РЭС-ТП. Вкладка подсвечивается загрузчику точкой до нажатия «Просмотрено».
- **Задания на участки**: приоритеты 1–3, создание вручную или пачкой из мёртвых ТП,
  автозакрытие при появлении опроса в новой загрузке.
- **Экспорт в Excel** на каждой вкладке; сводка — с диаграммами.

## Роли

| Роль | Права |
|---|---|
| `admin` | всё: загрузка, удаление загрузок и базы, пользователи |
| `uploader` | всё, кроме удаления базы и управления пользователями |
| `staff` (служба учёта) | видит всё, не загружает, выдаёт задания на РЭС |
| `res` (участок) | видит только свой РЭС, выгружает отчёты, закрывает задания |

Первый вход: `admin` / пароль из переменной `ADMIN_PASSWORD` (по умолчанию `admin` —
поменяйте сразу).

## Деплой на Амверу

1. Залейте репозиторий, создайте проект с окружением **Docker** (Dockerfile в корне).
2. Порт 80, постоянный диск смонтируйте в `/data` (там SQLite и копии загрузок) —
   `amvera.yml` уже это описывает.
3. Переменные окружения: `ADMIN_PASSWORD`, при желании `MAX_UPLOAD_MB` (по умолчанию 50).
4. В оболочке Платформы добавьте приложение в `src/config/apps.js` с URL сервиса.

## Структура

```
app/
  main.py      — FastAPI, все маршруты API
  db.py        — модели SQLite (users, uploads, meters, meter_states, changes, tasks)
  parser.py    — потоковый разбор xlsx, классификация производителей и маршрутов
  importer.py  — импорт загрузки, дифф с предыдущей, автозакрытие заданий
  analytics.py — все отчёты
  auth.py      — сессии и роли
  export.py    — Excel-отчёты (xlsxwriter, диаграммы)
static/        — SPA-фронт (vanilla JS + Chart.js)
```

## Заметки по данным

- Ключ ПУ = «Тип ПУ + серийный номер». Дубли строк внутри одного файла игнорируются
  (берётся первая).
- ТП — колонка **L** выгрузки (K — фидер).
- Обработка файла идёт в фоне ~1.5–2 мин на 130 тыс. строк; интерфейс сам опрашивает статус.
- «Отремонтировано»: на той же точке учёта (колонка H) старый ПУ пропал, появился новый
  и он собирается.
